# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 프로젝트 개요

식품의약품안전처 식품영양성분DB(가공식품)를 **중심 데이터**로, 질병관리청 국민건강영양조사(KNHANES)를 **보조 데이터**로 결합한 만성질환(당뇨·고혈압·비만) 식이 분석 웹대시보드. 데이터를 관리하는 **관리자(영양사·데이터 담당) 뷰**와 식이 정보를 조회하는 **사용자 뷰**로 분리된다.

최종 산출물은 `만성질환_식이분석_대시보드.html` — 외부 의존성 없이 더블클릭으로 실행되는 **단일 HTML 파일**(약 7MB, 데이터 임베드)이다.

## 빌드/실행/테스트

이 프로젝트에는 패키지 매니저·빌드 도구·테스트 러너·린터가 없다. 워크플로우는 다음과 같다.

- **실행**: `만성질환_식이분석_대시보드.html`을 브라우저에서 직접 연다(서버 불필요).
- **데이터 재생성**: Python + `pyreadstat`으로 `raw data/`의 SAS(zip)·CSV를 처리해 `data/*.json`을 만든다. 파이프라인 상세는 [재현용_프롬프트_명세.md](재현용_프롬프트_명세.md) §2 참조. 정부 도메인/CDN이 샌드박스에서 차단되므로 원본 데이터는 사용자가 직접 확보해 `raw data/`에 제공해야 한다.
- **검증**(HTML 저장 후 반드시): 파일 끝이 `render();</script></body></html>`로 닫히는지, `<script>`/`</script>` 개수가 일치하는지, 임베드된 JSON이 파싱되는지 확인한다. 6~7MB 단일 HTML은 저장 시 **파일 끝 잘림 사고**가 잦으므로, 대용량 쓰기는 한 번에 기록하고 저장 직후 위 항목을 점검할 것.

## 아키텍처

### 두 개의 독립 데이터 계층

가공식품 DB는 원재료·음식이 없는 **부분 데이터(가공식품 5만 행)**라 KNHANES 재료값에 덮어쓰기 부적합 → 별도 계층으로 결합한다.

- **`const DATA`** (전역) ← `data/dashboard_data.json`: KNHANES. 키 `{dishes, foods, clusters, person, meta}`. `person`은 성별×연령대 **가중 분위수**(음식조사 가중치 `wt_ntr` 적용 — 단순평균 금지) 통계.
- **`const PRODUCTS`** (전역) ← `data/products_data.json`: 식약처 식품DB. 키 `{schema, products, queue, clusters, std, stat, meta}`. `products`는 컬럼명 배열 `schema`를 인덱스로 참조하는 **행 배열**이다. 접근은 반드시 `pget(row, key)` 헬퍼를 통해서 한다(`data/products_view.js` 참조).

### 렌더링 모델 (프레임워크 없음)

- 전역 상태: `role`(user/admin), `tab`. `TABS` 상수가 역할별 탭 목록을 정의(`만성질환_식이분석_대시보드.html:147`).
- `render()`가 `tab`에 해당하는 뷰 함수(`vProd`, `vAover`, `vCrit`, `vIns`, …)를 호출해 `#main.innerHTML`을 통째로 교체하고, `post_<tab>()`가 있으면 후처리(이벤트/차트 재바인딩)한다(`:155`).
- 뷰 함수 명명 규칙: `v<Name>()`은 HTML 문자열 반환, `post_<name>()`은 후처리. 새 탭 추가 시 `TABS`·`render()`의 디스패치 맵·`v<Name>`을 함께 추가한다.
- **차트는 자체 SVG 함수**(`svgBars` 다계열 막대, `svgLine` 라인, `svgDonut` 도넛)로 그린다. 외부 라이브러리·CDN 절대 금지.

### 스크립트 로드 순서 (중요)

데이터를 임베드하는 `<script>`가 **가장 마지막**에 와서 그 안에서 `render()`를 호출한다. 앞선 로직 스크립트의 `render()` 호출은 `typeof PRODUCTS==='undefined'` 가드로 무해하게 통과한다(`:279`). 순서를 바꾸면 초기 렌더가 깨진다.

### 관련 파일 관계

- `dashboard_template.html` — `DATA`만 담던 **구버전 템플릿**(`/*__DATA__*/` 자리표시자, PRODUCTS 계층 없음). 최종 산출물이 아니다.
- `data/products_view.js` — 가공식품 계층 뷰 로직(`ptag`, `vProd`, `vPqueue`, `vPdup` 등)의 소스 모듈. 최종 HTML에는 이 내용이 인라인되어 있다.
- 편집은 원칙적으로 최종 `만성질환_식이분석_대시보드.html`에 직접 한다.

## 태그 판정 로직 (도메인 핵심)

판정은 **표준 1회 제공량 기준**으로 한다(1일 기준치를 쓰면 대다수 음식이 '제한'으로 몰려 변별력이 사라지기 때문 — 근거는 [재현용_프롬프트_명세.md](재현용_프롬프트_명세.md) §8). 1회 제공량은 제품명 키워드로 식약처 별지3 식품군에 분류해 부여하고, 미분류 식품은 100g/100ml 기준으로 폴백한다. 태그는 3단계(적합/주의/제한).

두 모드로 분리된다:

- **환자 모드**: 질환별 제한 영양소를 고정 임계로 판정(고혈압→나트륨, 당뇨→당류, 비만→에너지, 확장: 신장→칼륨). 임계는 관리자 뷰 '태그 기준 관리'(`vCrit`)에서 편집·버전관리·영향 시뮬레이션 가능.
- **일반인 모드**: 성별·연령 프로필로 KDRI 2025 하루 기준(EER·CDRR 등)을 계산해, 1회 제공량이 하루 기준의 **15% 이상=주의, 30% 이상=제한**으로 판정(`kdriEER`, `kdriDaily`, `kdriTags`).

기준 수치(별지3 1회 제공기준량, 식약처 1일 영양성분 기준치, KDRI EER/AMDR/CDRR)의 전체 표는 [재현용_프롬프트_명세.md](재현용_프롬프트_명세.md) §3–4에 있다. **없는 수치를 만들지 말 것 — 원문 표에서 확인된 값만 사용**한다.
