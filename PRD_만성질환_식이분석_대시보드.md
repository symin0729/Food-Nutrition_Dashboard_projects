# PRD: 만성질환 맞춤 식이 분석 웹대시보드

| 항목 | 내용 |
|---|---|
| 문서 버전 | v0.1 (초안) |
| 작성일 | 2026-07-08 |
| 작성자 | 민서영 |
| 상태 | 검토 대기 |

---

## 1. 개요 (Overview)

### 1.1 제품 요약
식품영양성분DB와 질병관리청 국민건강영양조사(KNHANES) 데이터를 결합하여, 당뇨·고혈압·비만 등 만성질환 기준으로 음식 데이터를 태깅·검수·탐색할 수 있는 웹대시보드. 데이터를 관리하는 영양사·데이터 담당자와 식이 정보를 조회하는 일반 사용자 양측의 편의성을 높이는 것을 목적으로 하는 실사용 도구이다.

### 1.2 문제 정의
- 공공 식품영양성분 데이터는 출처(식약처, 농촌진흥청 등)가 분산되어 있고, 동일 음식이 서로 다른 명칭·단위로 중복 수록되어 있어 그대로 서비스에 쓰기 어렵다.
- 영양성분 값은 100g 기준으로 제공되는 경우가 많아, 실제 사용자가 이해할 수 있는 1회 제공량·1인분 단위로의 변환과 검수가 필요하다.
- 만성질환자가 "이 음식을 먹어도 되는가"를 판단하려면 질환별 제한 영양소 기준으로 음식을 해석해 주는 계층이 필요하지만, 원본 DB에는 이러한 태그가 없다.
- 데이터 담당자는 검수 대상(결측, 이상치, 중복)을 수작업으로 찾아야 하며, 판정 기준 변경이 데이터 전체에 미치는 영향을 사전에 확인할 수단이 없다.

### 1.3 목표 (Goals)
1. 데이터 담당자가 표준화·검수·태깅 업무를 한 화면에서 수행할 수 있는 관리 도구 제공
2. 규칙 기반 자동 판정과 영양사 수동 검수를 결합한 신뢰 가능한 질환별 식이 태그 체계 구축
3. 일반 사용자가 음식을 검색하면 질환별 적합도와 국민 평균 대비 자기 섭취 수준을 즉시 확인할 수 있는 조회 환경 제공

### 1.4 비목표 (Non-Goals)
- 의학적 진단·처방 기능 (본 도구는 정보 제공 목적이며 의료행위를 대체하지 않음)
- AI 음식 이미지 인식 (향후 연동 대상으로만 명시, 본 PRD 범위 외)
- 모바일 네이티브 앱 (웹 반응형까지만 대응)

## 2. 사용자 정의 (Personas)

### 2.1 관리자: 영양사 / 데이터 담당자
- 업무: 음식명 표준화, 중복 정리, 영양성분 검수, 질환별 태그 기준 관리
- 니즈: 검수 대상이 자동으로 모이는 작업 큐, 기준 변경 시 영향 범위 사전 확인, 판정 근거 추적
- 성공 조건: 건당 검수 시간 단축, 수작업 탐색 제거

### 2.2 일반 사용자: 만성질환자 및 식단 관리자
- 상황: 당뇨·고혈압·비만(향후 신장질환 등) 진단을 받았거나 예방 목적으로 식단을 관리
- 니즈: 음식 검색 시 "먹어도 되는지"를 1회 제공량 기준으로 즉시 판단, 제한 음식의 대체재 확인, 내 섭취 수준의 객관적 비교
- 성공 조건: 검색→적합도 확인까지 3클릭 이내, 결과 없는 검색 최소화

## 3. 데이터 소스 및 요구사항

### 3.1 식품영양성분DB (식약처·농촌진흥청 통합)
- 출처: 공공데이터포털 「전국통합식품영양성분정보 표준데이터」 및 식약처 식품영양성분DB (로그인 없이 CSV/OpenAPI 제공)
- 주요 필드: 식품코드(17자리, 분류정보 포함), 식품명, 식품분류(원재료성식품/가공식품/음식), 영양성분함량기준량, 에너지(kcal), 탄수화물(g), 당류(g), 단백질(g), 지방(g), 포화지방산(g), 트랜스지방산(g), 콜레스테롤(mg), 나트륨(mg), 칼륨(mg), 인(mg), 식이섬유(g), 1회섭취참고량, 식품중량, 출처명, 데이터생성방법, 데이터기준일자
- 활용: 대시보드의 마스터 음식 DB. 100g 기준 → 1회 제공량/1인분/1개 단위 변환의 원천

### 3.2 국민건강영양조사 (질병관리청, KNHANES)
- 출처: knhanes.kdca.go.kr 원시자료 (무료 공개, 회원가입·이용약관 동의 후 다운로드 — 담당자가 직접 확보하여 업로드)
- 구성: 건강설문조사·검진조사·영양조사. 영양조사는 24시간 회상법 기반 식품섭취량 데이터 포함
- 주요 활용 변수: 성별, 연령, 유병 여부(당뇨·고혈압 등 검진/설문 변수), 영양소별 1일 섭취량
- 활용: (1) 연령·성별·질환군별 국민 평균 섭취량 산출 → 사용자 비교 기능, (2) 태그 기준 설정의 근거 데이터 → 관리자 인사이트

### 3.3 판정 기준 근거 (공식 기준)
- 「식품 등의 표시·광고에 관한 법률 시행규칙」 별표 5: 1일 영양성분 기준치 (나트륨 2,000mg, 당류 100g, 포화지방 15g, 콜레스테롤 300mg 등)
- 「식품등의 표시기준」 영양강조표시 기준: 저나트륨(100g당 120mg 이하), 저당류(100g당 5g 이하 / 100mL당 2.5g 이하), 무당류(100g당 0.5g 미만) 등
- WHO 나트륨 권고(2,000mg/일)를 보조 근거로 병기
- 기준치는 하드코딩하지 않고 관리자 뷰의 "태그 기준 관리"에서 버전 관리되는 설정값으로 운영

### 3.4 데이터 파이프라인 요구사항
1. 수집: CSV/API 적재 (식품DB), 원시자료 파일 업로드 (KNHANES)
2. 정제: 결측·이상치 플래깅, 단위 통일, 인코딩 정규화
3. 표준화: 음식명 유사군 클러스터링 → 대표명 매핑, 카테고리 분류
4. 변환: 100g 기준 → 1회 제공량 환산 (1회섭취참고량·식품중량 필드 활용)
5. 태깅: 질환별 규칙 엔진 자동 판정 → 검수 큐 → 영양사 승인 시 확정

## 4. 기능 요구사항 (Functional Requirements)

우선순위 표기: P0(필수) / P1(중요) / P2(후순위)

### 4.1 관리자 뷰 — Phase 1 (MVP)

#### FR-A1. 영양성분 검수 큐 (P0)
- 자동 플래깅 규칙: 필수 영양소 결측, 단위 이상치(예: 100g당 나트륨 10,000mg 초과), 에너지-영양소 계산 불일치(탄수화물×4 + 단백질×4 + 지방×9 대비 ±20% 초과), 1회 제공량 미기재
- 검수 상태 워크플로우: 대기 → 검수중 → 승인 / 반려(수정 요청)
- 항목별 원본값·수정값·검수자·검수일시 이력 저장
- 필터: 플래그 유형, 식품분류, 출처, 상태

#### FR-A2. 음식명 표준화·중복 관리 (P0)
- 문자열 유사도 기반 유사 음식명 후보군 자동 제시 (예: "김치찌개 / 김치 찌개 / 김치찌게")
- 대표명 지정 및 하위 명칭 매핑, 병합 이력 관리
- 카테고리 분류 현황판 (미분류 항목 수 추적)

#### FR-A3. 식이 태그 기준 관리 (P0)
- 질환별 제한 영양소·임계값을 설정하는 규칙 편집기
  - 고혈압: 나트륨 (1회 제공량 기준)
  - 당뇨: 당류, 탄수화물
  - 비만: 에너지, 지방, 포화지방
  - (확장) 신장질환: 칼륨, 인, 단백질 — 스키마에 질환·영양소를 추가 가능한 구조로 설계
- 태그 3단계: 적합 / 주의 / 제한
- 기준 변경 시 시뮬레이션: 저장 전 "이 기준으로 태그가 바뀌는 음식 N건" 미리보기
- 기준 버전 관리 (변경자, 변경일, 근거 메모)

#### FR-A4. 국민건강영양조사 인사이트 (P1)
- 연령대·성별·유병군별 영양소 섭취 분포 시각화 (박스플롯/히스토그램)
- 태그 기준값과 실제 섭취 분포의 중첩 표시 → 기준 설정 근거 자료
- 조사 연도 선택 및 연도 간 추이 비교

### 4.2 사용자 뷰 — Phase 2

#### FR-U1. 음식 검색 + 질환별 적합도 (P0)
- 표준화된 대표명 기반 검색 (동의어 매핑으로 검색 실패 최소화)
- 결과 카드: 1회 제공량 기준 영양성분 + 질환별 적합/주의/제한 태그 + 판정 근거(어떤 영양소가 기준 대비 몇 %인지)
- 검수 승인된 데이터만 노출 (미검수 데이터는 사용자 뷰에서 제외)

#### FR-U2. 대체 식품 추천 (P1)
- "제한" 태그 음식 조회 시 동일 카테고리 내 "적합" 태그 음식을 제한 영양소 오름차순으로 제시

#### FR-U3. 내 섭취 vs 국민 평균 비교 (P1)
- 사용자가 연령·성별 입력 후 섭취 음식/양 입력 → KNHANES 동일 집단 평균·분포 대비 백분위 시각화

#### FR-U4. 일일 식단 기록·분석 (P2)
- 하루 식단 누적 기록, 질환 기준 대비 초과 영양소 경고
- Phase 2 후반 또는 Phase 3로 이월 가능

### 4.3 공통·시스템 (P0)
- 역할 기반 접근 제어: 관리자 / 일반 사용자
- 데이터 출처·기준일자 화면 표기 (신뢰성)
- 의료 면책 고지 문구 상시 노출

## 5. 화면 구성 (Information Architecture)

```
대시보드
├─ [관리자] 홈: 검수 대기 건수, 미분류 건수, 데이터 품질 요약 KPI
│   ├─ 검수 큐 (FR-A1)
│   ├─ 표준화·중복 관리 (FR-A2)
│   ├─ 태그 기준 관리 (FR-A3)
│   └─ KNHANES 인사이트 (FR-A4)
└─ [사용자] 홈: 음식 검색창 + 질환 프로필 선택
    ├─ 검색 결과·적합도 상세 (FR-U1)
    ├─ 대체 식품 추천 (FR-U2)
    ├─ 국민 평균 비교 (FR-U3)
    └─ 식단 기록 (FR-U4)
```

## 6. 단계별 출시 계획 (Release Plan)

| Phase | 범위 | 완료 기준 |
|---|---|---|
| Phase 1 (MVP) | 데이터 관리 층: FR-A1, FR-A2, FR-A3 + 파이프라인 수집·정제 | 식품DB 전량 적재, 자동 플래깅 동작, 태그 기준 1차 확정 |
| Phase 2 | 사용자 층: FR-U1, FR-U2, FR-U3 + FR-A4 | 검수 승인 데이터만 노출되는 검색·적합도 화면 오픈 |
| Phase 3 | FR-U4 식단 기록, 신장질환 등 질환 확장, AI 음식 인식·외부 영양 API 연동 검토 | 확장 질환 1종 이상 태그 운영 |

데이터 관리 층을 선행하는 이유: 검수되지 않은 데이터가 사용자에게 노출되는 위험을 차단하고, 사용자 기능의 품질이 데이터 품질에 종속되기 때문이다.

## 8. 기술 고려사항 (Technical Considerations)

- 프론트엔드: 웹 반응형 대시보드 (구현 스택은 프로토타입 단계에서 확정)
- 데이터 저장: 원본 테이블과 검수·표준화 결과 테이블 분리 (원본 불변 원칙), 태그 기준 테이블은 버전 컬럼 포함
- 태그 판정: 규칙 엔진 방식 — 기준 테이블 변경만으로 재판정 가능해야 하며 코드 수정 불필요
- KNHANES 통계 산출 시 원시자료 이용지침서의 가중치(복합표본설계) 적용 필요 — 단순 평균 사용 금지
- 개인 식단 기록 데이터는 개인정보 최소 수집 원칙 적용

## 9. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| 두 데이터의 음식 분류 체계 불일치 (식품DB 식품코드 vs KNHANES 식품코드) | 비교 기능 정확도 저하 | Phase 1에서 매핑 테이블 구축, 매핑 불가 항목은 카테고리 수준 비교로 대체 |
| 1회 제공량 정보 결측 | 태그 판정 불가 | 결측 시 100g 기준 판정으로 폴백하고 화면에 기준 단위 명시 |
| 질환 기준의 의학적 논쟁 여지 | 신뢰도 문제 | 공식 고시 기준만 사용, 근거 출처를 태그 상세에 표기, 면책 고지 |
| KNHANES 원시자료 확보 지연 | FR-A4, FR-U3 지연 | 해당 기능을 P1로 분리하여 MVP 경로에서 제외 |

## 10. 미해결 질문 (Open Questions)

1. 사용자 계정 체계: 로그인 없는 게스트 조회를 허용할 것인가, 식단 기록은 계정 필수인가
2. 질환 복수 보유 사용자(예: 당뇨+고혈압)의 적합도 표시 방식 — 가장 엄격한 태그 우선 표시안 검토
3. 배달음식·편의점 식품 등 상용 제품 데이터의 갱신 주기 정책
