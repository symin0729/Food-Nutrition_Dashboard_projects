## 2. 데이터 처리 파이프라인 (Python + `pyreadstat`)

### 2.1 KNHANES ALL — 인구·질환·영양소 가중 통계
- 연도별로 압축 해제 → 필요 변수만 추출 → 원본 삭제(디스크 절약).
- 추출 변수: `ID, year, sex, age, kstrata, psu, wt_ntr, HE_HP, HE_BMI, HE_glu, HE_HbA1c, HE_DM_HbA1c, DI1_dg, DE1_dg` + `N_`로 시작하는 영양소(`N_EN, N_CHO, N_SUGAR, N_FAT, N_SFA, N_PROT, N_NA, N_K, N_PHOS, N_CHOL, N_TDF …`).
- 질환군: 고혈압 `HE_HP==3 or DI1_dg==1`, 당뇨 `DE1_dg==1 or HE_DM_HbA1c==3`, 비만 `HE_BMI>=25`.
- 만 19세 이상 성인 & 영양조사 완료자만(약 26,363명).
- **가중치 `wt_ntr` 적용** 가중 분위수(5% 간격): 성별×연령대(19-29/30-39/40-49/50-59/60-69/70+) 분포, 질환 유병/비유병 평균, 연도 추이. **단순평균 금지.**

### 2.2 KNHANES 24RC — 음식·식재료 마스터
- 인코딩 오류 시 `encoding='cp949'` 폴백. 대용량은 `row_offset/row_limit` 분할 후 gzip 병합.
- 음식(dish): `N_DCODE`(코드)·`N_DNAME`(명). 1회 섭취분 = `(ID, N_MEAL, N_DCODE)` 그룹의 `NF_*` 합. 관찰 10회 이상만, 값은 중앙값.
- 식재료(food): `N_FCODE`·`N_FNAME`, 섭취량(`NF_INTK`)으로 나눠 **100g 기준** 중앙값.
- 산출 `dashboard_data.json` = `{ dishes(≈3,021), foods(≈1,922), clusters(중복명 ≈454), person(가중 통계), meta }`.

### 2.3 식품영양성분DB — 중심 데이터
- cp949 파싱. 기준량 100g 또는 100ml(`isml` 플래그). 컬럼 매핑: 에너지(kcal)/나트륨(mg)/당류(g)/탄수화물(g)/단백질(g)/지방(g)/포화지방산(g)/콜레스테롤(mg)/칼륨(mg)/칼슘(mg)/식이섬유(g).
- 검수 플래그(100g 기준): 에너지 불일치 `|4·단백+4·탄수+9·지방 − 에너지|/에너지 > 30%`, 나트륨 `>3,000mg/100g`, 당류 `>80g/100g`, 필수값 결측.
- 중복 클러스터: 정규화 명칭(공백·괄호·기호 제거) 동일·식품코드 상이.
- 사용자 검색용: 동일 제품명을 중앙값 1행으로 축약 → 고유명 ≈35,349종.
- 산출 `products_data.json` = `{ schema(컬럼명 배열), products(행 배열), queue, clusters, std, stat, meta }`. **행 접근은 `schema` 인덱스 기반(`pget(row,key)`).**

## 4. 태그 판정 로직 (핵심)

판정 단위는 **표준 1회 제공량**(1일 기준치를 쓰면 대다수 음식이 '제한'으로 몰려 변별력 상실 — §7 근거). 태그 3단계: 적합/주의/제한. 두 모드로 분리.

### 4.1 환자 모드 (질환별 고정 제한, 치료식 관점)
- 고혈압 → 나트륨: 주의 300mg / 제한 600mg
- 당뇨 → 당류: 주의 10g / 제한 25g
- 비만 → 에너지: 주의 400kcal / 제한 700kcal
- (확장) 신장질환 → 칼륨: 주의 450mg / 제한 900mg
- 임계는 관리자 뷰 '태그 기준 관리'에서 편집·버전관리·영향 시뮬레이션.

### 4.2 일반인 모드 (KDRI 2025, 프로필별 가변)
사용자가 성별·연령 선택 → 하루 기준 계산: 에너지=EER, 나트륨=CDRR, 당류=`0.20·EER/4`, 포화지방=`0.07·EER/9`, 콜레스테롤=300mg. 표준 1회 제공량의 각 영양소가 **하루 기준의 15% 이상=주의, 30% 이상=제한**, 미만=적합.

## 5. 대시보드 사양

- 단일 HTML, 외부 스크립트/CDN 미사용. 그래프는 자체 SVG 함수(`svgBars` 다계열 막대, `svgLine` 라인, `svgDonut` 도넛).
- 전역 데이터: `const DATA`(KNHANES) ← `dashboard_data.json`, `const PRODUCTS`(식품DB) ← `products_data.json`.
- 렌더링: 전역 `role`(user/admin)·`tab`, `TABS` 상수가 역할별 탭 정의, `render()`가 뷰 함수(`v<Name>`) 호출로 `#main.innerHTML` 교체 후 `post_<tab>()` 후처리.
- **스크립트 순서:** 데이터 임베드 `<script>`가 가장 마지막에 와서 그 안에서 `render()` 호출. 앞선 `render()`는 `typeof PRODUCTS==='undefined'` 가드로 무해 통과.
- 초기 화면: 관리자 뷰·개요 탭(상단 토글로 사용자 뷰 전환).

**관리자 탭(7):** 개요(품질 도넛·질환별 태그 분포 막대) / 영양성분 검수(플래그 큐·도넛·태그 막대) / 표준화·중복(대표코드 지정) / 태그 기준 관리(임계 편집·시뮬레이션·버전·별지3·KDRI 참고표) / KNHANES 인사이트(질환·성연령·연도추이) / KNHANES 음식 검수 / KNHANES 음식 표준화.

**사용자 탭(4):** 식품영양성분 검색(일반인/환자 모드 토글·적합도·상세·대체식품) / 국민 평균 비교(동일 성·연령 분위수 대비 내 섭취) / KNHANES 음식 검색 / 식단 기록(누적·1일 기준 대비 경고·CSV 내보내기).

**하단 상시 고지:** 정보 제공 목적, 의학적 진단·처방 대체 아님.

## 8. 재현 시 반드시 지킬 것

- **거대 단일 HTML(6~7MB) 저장 시 파일 끝 잘림 주의.** 대용량 쓰기는 파이썬 `open().write()` 한 번에 기록하고, 저장 후 `</html>` 종료·`<script>`/`</script>` 개수 일치·JSON 파싱을 검증.
- 데이터 임베드 스크립트가 **가장 마지막**, 그 안에서 `render()` 호출(앞선 `render()`는 `typeof PRODUCTS==='undefined'` 가드).
- KNHANES 통계는 **가중치 필수**(단순평균 금지).
- **없는 수치 생성 금지** — KDRI/별지 값은 원문 표 확인분만.
- 문체: 두괄식 학술 보고서(목록·과장대비·정서적 형용사 지양).

## 9. 설계 근거 (데이터에서 확인된 사실)

- KNHANES 남 50대 나트륨 섭취 중앙값 ≈3,678mg으로 1일 기준치(2,000mg)의 1.8배 → 1일 기준치를 태그 임계로 쓰면 변별력 저하 → **1회 제공량 기준** 채택.
- 식품DB는 원재료·음식 없이 **가공식품 5만 행(부분)** → KNHANES 재료값 덮어쓰기 부적합 → **독립 계층**으로 결합.
- 라면(표준 120g, 100g당 나트륨 1,240mg) 1회 섭취 시 1,488mg → 남30-49(CDRR 2,300)의 ≈65%, 여75+(1,800)의 ≈83% → 프로필별 태그 가변(일반인 모드 핵심 동작).
