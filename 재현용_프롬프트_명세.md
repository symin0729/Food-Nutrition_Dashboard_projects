# 만성질환 식이 분석 대시보드 — 재현용 프롬프트 & 명세

이 문서 하나만 보고도 동일한 산출물(기획 PRD + 단일 HTML 대시보드)을 재현할 수 있도록, 목표·입력데이터·처리 파이프라인·적용 기준·대시보드 사양·단계별 지시를 정리한다.

---

## 0. 최종 산출물

- `PRD_만성질환_식이분석_대시보드.md` — 기획 문서(PRD)
- `만성질환_식이분석_대시보드.html` — 외부 의존성 없는 단일 파일 대시보드(더블클릭 실행, 서버 불필요)
- `data/` — 정제 데이터(`dashboard_data.json`, `products_data.json`, KNHANES 정제 CSV)

핵심 성격: **식품의약품안전처 식품영양성분DB(가공식품)를 중심 데이터**, **질병관리청 국민건강영양조사(KNHANES)를 보조 데이터**로 결합한 만성질환 식이 분석 도구. 관리자(영양사·데이터 담당) 뷰와 사용자 뷰를 분리.

## 2. 처리 파이프라인

### 2.1 KNHANES ALL (인구·질환·영양소 통계)
- 라이브러리: `pyreadstat`(sas7bdat 읽기). 파일별 압축 해제 후 아래 변수만 추출(디스크 절약 위해 연도별 처리 후 원본 삭제).
- 추출 변수: `ID, year, sex, age, kstrata, psu, wt_ntr, HE_HP, HE_BMI, HE_glu, HE_HbA1c, HE_DM_HbA1c, DI1_dg, DE1_dg` + `N_`로 시작하는 영양소(N_EN, N_CHO, N_SUGAR, N_FAT, N_SFA, N_PROT, N_NA, N_K, N_PHOS, N_CHOL, N_TDF …).
- 질환군 정의: 고혈압 `HE_HP==3 or DI1_dg==1`, 당뇨 `DE1_dg==1 or HE_DM_HbA1c==3`, 비만 `HE_BMI>=25`.
- 성인(age>=19)·영양조사 완료자만 사용(약 26,363명).
- **가중치 wt_ntr 적용**한 가중 분위수(5% 간격) 산출: 성별×연령대(19-29/30-39/40-49/50-59/60-69/70+)별 영양소 분포, 질환군 유병/비유병 평균, 연도 추이.

### 2.2 KNHANES 24RC (음식·식재료 마스터)
- 인코딩 오류 시 `encoding='cp949'` 폴백. 대용량(파일당 60만 행) → `row_offset/row_limit`로 분할 처리 후 gzip 병합.
- 음식(dish) 단위: `N_DCODE`(음식코드), `N_DNAME`(음식명). 섭취 1회분 = (ID, N_MEAL, N_DCODE) 그룹의 NF_* 합. 음식별 관찰 10회 이상만, 값은 중앙값.
- 식재료(food) 단위: `N_FCODE`, `N_FNAME`, 섭취량(NF_INTK)으로 나눠 **100g 기준** 중앙값.
- 산출: `dashboard_data.json` = { dishes(3,021), foods(1,922), clusters(중복명 454), person(가중 통계) }.

### 2.3 식품영양성분DB (중심 데이터)
- cp949 파싱. 컬럼 매핑: 에너지(kcal)/나트륨(mg)/당류(g)/탄수화물(g)/단백질(g)/지방(g)/포화지방산(g)/콜레스테롤(mg)/칼륨(mg)/칼슘(mg)/식이섬유(g). 기준량 100g 또는 100ml(`isml` 플래그).
- 검수 플래그(품질=100g 기준): 에너지 불일치 `|4·단백+4·탄수+9·지방 − 에너지|/에너지 > 30%`, 나트륨 `>3000mg/100g`, 당류 `>80g/100g`, 필수값 결측.
- 중복 클러스터: 정규화 명칭(공백·괄호·기호 제거) 동일·식품코드 상이.
- 사용자 검색용: 식품명 대표(중앙값) 1행으로 축약 → 고유명 35,349종.
- 산출: `products_data.json` = { schema(배열), products, queue, clusters, std, stat }.

## 4. 태그 판정 로직 (핵심)

두 모드로 분리(사용자 요구: **일반인이면 KDRI 전면 적용, 환자면 기존 질환 제한 유지**).

### 4.1 환자 모드 (질환별 제한, 치료식 관점)
질환별 제한 영양소를 **표준 1회 제공량 기준**으로 판정. 임계: 적합/주의/제한.
- 고혈압 → 나트륨: 주의 300mg, 제한 600mg (1일 2,000mg의 15%/30%)
- 당뇨 → 당류: 주의 10g, 제한 25g
- 비만 → 에너지: 주의 400kcal, 제한 700kcal
- (확장) 신장질환 → 칼륨: 주의 450mg, 제한 900mg
- 태그 기준은 관리자 뷰에서 편집·버전관리·영향 시뮬레이션 가능.

### 4.2 일반인 모드 (KDRI 2025, 프로필별 가변)
사용자가 성별·연령 선택 → 그 프로필의 하루 기준 계산:
- 에너지=EER, 나트륨=CDRR, 당류=`0.20·EER/4`, 포화지방=`0.07·EER/9`, 콜레스테롤=300mg.
- 판정: 표준 1회 제공량의 각 영양소가 **하루 기준의 15% 이상=주의, 30% 이상=제한**, 그 미만=적합.

## 5. 대시보드 사양

- 단일 HTML, 외부 스크립트/CDN 미사용. 그래프는 자체 SVG 함수(`svgBars` 다계열 막대, `svgLine` 라인, `svgDonut` 도넛).
- 초기 화면: **관리자 뷰 · 개요 탭**(상단 토글로 사용자 뷰 전환).
- 데이터는 HTML에 `const DATA`(KNHANES), `const PRODUCTS`(식품DB)로 임베드.

**관리자 탭(7):** 개요(식품DB 중심 KPI·품질 도넛·태그분포 막대) / 영양성분 검수(플래그 큐·도넛·태그 막대) / 표준화·중복(대표코드 지정) / 태그 기준 관리(임계 편집·시뮬레이션·버전·별지3·KDRI 참고표) / KNHANES 인사이트(질환·성연령·연도추이 그래프) / KNHANES 음식 검수 / KNHANES 음식 표준화.

**사용자 탭(4):** 식품영양성분 검색(일반인/환자 모드 토글·질환 또는 프로필별 적합도·상세·대체식품) / 국민 평균 비교(KNHANES 동일 성·연령 분위수 대비 내 섭취) / KNHANES 음식 검색 / 식단 기록(누적·1일 기준 대비 경고·CSV 내보내기).

**하단 상시 고지:** 정보 제공 목적, 의학적 진단·처방 대체 아님.

## 6. 재현 절차 (에이전트에게 줄 단계별 지시)

1. **PRD 작성**: "식품영양성분DB와 KNHANES를 결합한 만성질환 식이 분석 웹대시보드의 PRD를 두괄식 학술문서로 작성. 관리자·사용자 양측 편의 목적, 데이터관리(Phase1)→사용자기능(Phase2)→확장(Phase3), KPI·리스크·미해결질문 포함."
2. **KNHANES 처리**: "업로드한 HN20~24 ALL/24RC(SAS)를 연도별로 압축 해제→필요 변수만 추출(디스크 절약)→가중치(wt_ntr) 적용 통계와 음식·식재료 마스터를 만들어 `dashboard_data.json` 생성." (§2.1–2.2)
3. **식품DB 처리**: "`전국통합식품영양성분정보표준데이터.csv`(cp949)를 파싱해 100g 기준값·검수 플래그·중복 클러스터·대표행을 만들고 `products_data.json` 생성." (§2.3)
4. **대시보드 구축**: "외부 의존성 없는 단일 HTML 대시보드 생성. 관리자/사용자 뷰, §5 탭 구성, 자체 SVG 차트. 데이터 임베드." 
5. **1회 제공량 표준 반영**: "별지3 식품별 1회 제공기준량을 제품명 키워드 규칙으로 부여(미분류는 100g/ml), 태그를 표준 1회량 기준으로 판정." (§3.1)
6. **KDRI 2025 반영**: "일반인 모드=KDRI(EER·AMDR·CDRR, §3.3) 프로필별 가변(하루기준 15%/30%), 환자 모드=기존 질환 제한 유지. 관리자 태그 기준 관리에 KDRI 참고표 추가." (§4)

## 7. 주의점 (재현 시 반드시 지킬 것)

- **거대 단일 HTML(6~7MB) 저장 시 파일 끝 잘림 주의.** 대용량 쓰기는 파이썬 `open().write()`(한 번에 기록)로 하고, 저장 후 파일 마지막이 `</html>`로 끝나는지, `<script>`와 `</script>` 개수가 일치하는지 검증할 것.
- 데이터를 임베드하는 스크립트가 **가장 마지막**에 오고, 그 안에서 `render()`를 호출해야 함(앞선 `render()`는 `typeof PRODUCTS==='undefined'` 가드로 무해 통과).
- KNHANES 통계는 반드시 **가중치**를 적용(단순평균 금지).
- 없는 수치는 생성하지 말 것. KDRI/별지 값은 원문 표에서 확인된 것만 사용.
- 사용자 프로필: 학술 보고서 스타일(두괄식, 목록·과장대비 지양), 정서적 형용사 제거.

## 8. 데이터에서 확인된 사실 (설계 근거)

- KNHANES 성인 나트륨 섭취 중앙값이 남 50대 약 3,678mg으로 1일 기준치(2,000mg)의 1.8배 → 태그 임계를 1일 기준치 그대로 쓰면 변별력 저하 → **1회 제공량 기준** 채택.
- 식품DB는 원재료·음식 없이 **가공식품 5만 행(부분)** → KNHANES 재료값 덮어쓰기 부적합 → **독립 계층**으로 결합.
- 라면(표준 120g, 100g당 나트륨 1,240mg) 1회 섭취 시 1,488mg → 남30-49(CDRR 2,300) 약 65%, 여75+(1,800) 약 83% → 프로필별로 태그 가변.
